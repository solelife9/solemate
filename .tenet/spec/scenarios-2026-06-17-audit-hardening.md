# Scenarios: Audit Hardening

## Scenarios (Success)
1. **최신 우선 머지** — 폰A가 런 X 거리를 15→12km로 수정(updatedAt 갱신) 후 동기화. 폰B(로컬 15km, 더 오래된 updatedAt)가 동기화 → 머지가 폰A의 12km(최신)를 채택하고 폰B로 전파. 15km가 12km를 덮지 않는다.
2. **삭제 전파(tombstone)** — 폰A가 런 X 삭제. 폰B(오프라인, X 보유)가 온라인 후 동기화 → 합집합 머지가 X를 부활시키지 않고 tombstone을 존중해 양쪽에서 사라진다.
3. **오프라인 부팅 가시성** — 백엔드 다운 중 런 3개 추가(pending 큐 적재). 앱 강제종료 후 재실행(백엔드 여전히 다운) → 오프라인 부팅이 캐시 + pending을 오버레이해 3개 런이 UI에 보이고 신발 누적거리에 반영된다.
4. **런 햅틱 피드백** — 카운트다운 3·2·1·GO, 런 시작/일시정지/재개, 목표달성, 길게눌러종료 확정 시 `lib/haptics`의 해당 메서드가 호출된다.
5. **런플로우 theme 수렴** — RunActive/RunGoal/RunCountdown/Onboarding이 `theme.ts` 토큰만 사용. 사설 팔레트 객체·BebasNeue·하드코딩 hex(브랜드 SVG 제외)가 없다.
6. **삭제 undo** — 런/신발 삭제 시 "삭제됨 · 실행취소" 스낵바가 뜨고, 실행취소를 누르면 레코드가 (updatedAt 보존하며) 복원된다.
7. **타입 안전 데이터 경계** — `lib/api.ts`·`lib/stats.ts`가 `any` 없이 도메인 타입을 사용한다. 필드명 변경 시 컴파일 에러로 잡힌다.
8. **리스트 가상화** — 런 수백 개에서도 HistoryScreen이 `FlatList`로 가시영역만 렌더(전부 마운트하지 않음).
9. **단일 CTA** — 모든 주요 액션 버튼이 단일 `Button` 프리미티브를 쓰고 동일 radius/그라데이션을 가진다.
10. **그린 유지** — 각 묶음 완료 시 tsc/lint/jest 모두 green.

## Anti-Scenarios (Failure — 반드시 방지)
1. stale 로컬 레코드가 최신 원격 수정을 조용히 덮어쓴다(updatedAt 무시).
2. 한 기기서 지운 레코드가 다른 기기 동기화로 되살아난다.
3. 오프라인에서 추가한 런이 재부팅 후 UI에서 사라진다(데이터는 살아있어도 안 보임).
4. 신규 네이티브 의존성을 추가해 자율 검증 불가/Android 빌드 깨짐.
5. 사용자 신발/런 데이터가 마이그레이션 없이 파괴적으로 바뀐다(updatedAt 시드가 기존 값 손상).
6. 햅틱/토스트/포맷 변경이 거리·시간·페이스 계산 결과를 바꾼다(순수 로직 회귀).
7. theme 흡수 과정에서 런플로우 화면 색/폰트가 시각적으로 깨진다(다크+오렌지 정체성 훼손).
8. tsc/lint/jest 중 하나라도 red인 채로 묶음을 머지한다.
9. 삭제 undo가 부분 복원(런은 살아도 route_/time_/surface_ 사이드키 유실)된다.
10. FCM 토큰 배선이 실패 시 부팅을 막는다(graceful no-op이어야 함).
